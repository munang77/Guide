21ModuleScript — 코드 정리의 시작 고급

스크립트 한 파일이 500줄을 넘어가면 끝입니다. 기능별로 쪼개서 부품화합니다.

-- CoinManager (ModuleScript, ServerScriptService 안)
local CoinManager = {}

function CoinManager.give(player, amount)
    player.leaderstats.Coins.Value += amount
end

function CoinManager.canAfford(player, price)
    return player.leaderstats.Coins.Value >= price
end

return CoinManager   -- 반드시 무언가를 return
-- 다른 스크립트에서 사용
local CoinManager = require(game.ServerScriptService.CoinManager)
CoinManager.give(player, 50)
  • require는 한 번만 실행되고 캐시됩니다 — 여러 스크립트가 require해도 같은 테이블을 공유. 이 성질로 모듈이 공유 상태 저장소가 됩니다.
  • 서버 전용 모듈은 ServerScriptService, 서버·클라 공용은 ReplicatedStorage에.
  • 설계 기준: 한 모듈 = 한 책임 (CoinManager는 코인만, ShopManager는 상점만).

22OOP와 메타테이블 고급

"몬스터 100마리"를 만들 때 — 설계도(클래스) 하나로 인스턴스를 찍어내는 방법.

-- Monster (ModuleScript)
local Monster = {}
Monster.__index = Monster

function Monster.new(name, hp)
    local self = setmetatable({}, Monster)
    self.name = name
    self.hp = hp
    self.maxHp = hp
    return self
end

function Monster:takeDamage(amount)   -- 콜론 = self 자동 전달
    self.hp = math.max(0, self.hp - amount)
    if self.hp == 0 then self:die() end
end

function Monster:die()
    print(self.name .. " 사망")
end

return Monster
local Monster = require(RS.Monster)
local goblin = Monster.new("고블린", 50)
goblin:takeDamage(30)   -- hp 20

원리

  • 메타테이블은 "테이블에 없는 키를 찾으면 어디서 대신 찾을지" 정하는 장치. __index = Monster 덕분에 goblin에 없는 takeDamage를 Monster에서 찾아 씁니다.
  • : 호출은 goblin.takeDamage(goblin, 30)의 줄임 — self가 자동으로 들어갑니다.
  • 상속도 가능 (Boss의 __index를 Monster로) — 단, 깊은 상속은 디버깅 지옥. 1단계까지만 권장.

23DataStore — 데이터 영구 저장 고급

게임을 껐다 켜도 코인이 남아있게. 서버 전용이며, 실패할 수 있다는 전제로 짭니다.

local DSS = game:GetService("DataStoreService")
local store = DSS:GetDataStore("PlayerData_v1")
local Players = game:GetService("Players")

Players.PlayerAdded:Connect(function(player)
    local ok, data = pcall(function()
        return store:GetAsync("p_" .. player.UserId)
    end)
    local coins = (ok and data and data.coins) or 0
    -- leaderstats 생성 후 coins 적용...
end)

Players.PlayerRemoving:Connect(function(player)
    pcall(function()
        store:SetAsync("p_" .. player.UserId, {
            coins = player.leaderstats.Coins.Value,
        })
    end)
end)

반드시 지킬 것

  • pcall 필수: DataStore는 네트워크 호출 — 실패할 수 있습니다. pcall 없이 에러가 나면 스크립트 전체가 죽습니다.
  • 키는 UserId로: 닉네임은 바뀝니다.
  • BindToClose: 서버가 닫힐 때 마지막 저장 — game:BindToClose(saveAll).
  • UpdateAsync > SetAsync: 경쟁 상황(두 서버 동시 저장)에선 기존 값을 받아 갱신하는 UpdateAsync가 안전.
  • Studio에서 테스트하려면 Game Settings > Security > Enable Studio Access to API Services 켜기.
!!

저장 구조를 바꿀 땐 스토어 이름의 버전(_v1)을 올리거나 마이그레이션 코드를 — 옛 데이터와 새 코드가 충돌하면 유저 데이터가 날아갑니다.

24CFrame — 위치와 회전의 수학 고급

Position은 위치만, CFrame은 위치 + 바라보는 방향. 3D 조작의 만능 도구.

-- 기본
part.CFrame = CFrame.new(0, 10, 0)                  -- 위치
part.CFrame = CFrame.new(pos) * CFrame.Angles(0, math.rad(45), 0)  -- + Y축 45도
part.CFrame = CFrame.lookAt(pos, target.Position)    -- target을 바라보게

-- 곱셈 = "기준으로 상대 이동"
part.CFrame = part.CFrame * CFrame.new(0, 0, -5)   -- 자기가 보는 방향으로 5 전진
part.CFrame = part.CFrame * CFrame.Angles(0, math.rad(10), 0)  -- 제자리 10도 회전

핵심 직관

  • A * B는 "A의 입장에서 B만큼" — 순서가 중요합니다. 회전 먼저 곱하면 결과가 다릅니다.
  • cf.LookVector = 앞 방향, RightVector, UpVector — "앞으로 쏘기"는 pos + cf.LookVector * 50.
  • cf:ToObjectSpace(other) / ToWorldSpace — 상대 좌표 변환. 탈것에 붙은 포탑 같은 것.
  • 보간: cf1:Lerp(cf2, 0.1) — 매 프레임 10%씩 따라가는 부드러운 추적 카메라.
-- 실전: 플레이어 주위를 도는 오브젝트
local angle = 0
RunService.Heartbeat:Connect(function(dt)
    angle += dt * 2
    orb.CFrame = root.CFrame
        * CFrame.Angles(0, angle, 0)
        * CFrame.new(5, 2, 0)      -- 반경 5, 높이 2
end)

25Raycast — 보이지 않는 광선 고급

총알 판정, 바닥 감지, 시야 체크 — "이 방향에 뭐가 있나"를 묻는 도구.

local params = RaycastParams.new()
params.FilterType = Enum.RaycastFilterType.Exclude
params.FilterDescendantsInstances = {character}   -- 자기 몸은 제외

local origin = root.Position
local direction = root.CFrame.LookVector * 100   -- 앞으로 100

local result = workspace:Raycast(origin, direction, params)
if result then
    print("맞은 것:", result.Instance.Name)
    print("위치:", result.Position)
    print("면 방향:", result.Normal)
    print("거리:", result.Distance)
end

활용 패턴

  • 총/마법 명중: 발사 위치에서 조준 방향으로 — result.Instance의 부모에서 인스턴스. 체력·이동속도·점프·사망 담당">Humanoid를 찾아 데미지.
  • 바닥 감지: 발 밑으로 짧게 (0,-5,0) — result.Instance.Material로 "잔디 위에선 발소리 다르게".
  • 시야 체크: 몬스터→플레이어 레이가 벽에 막히면 "안 보임".
  • 벽 통과 배치 방지: 설치 위치까지 레이를 쏴 막히는지 확인.
  • Normal 방향으로 이펙트를 정렬하면(CFrame.lookAt(pos, pos + normal)) 벽에 딱 붙는 총알 자국.
TIP

장식 파트의 CanQuery를 false로 해두면 레이가 무시합니다 — 맵 가이드의 충돌 설정과 만나는 지점입니다.

26CollectionService와 Attribute 고급

"용암 파트 200개에 스크립트 200개"를 끝내는 법 — 태그 하나, 스크립트 하나.

태그

local CS = game:GetService("CollectionService")

local function setupLava(part)
    part.Touched:Connect(function(hit)
        local h = hit.Parent:FindFirstChildOfClass("Humanoid")
        if h then h.Health = 0 end
    end)
end

for _, part in ipairs(CS:GetTagged("Lava")) do
    setupLava(part)            -- 이미 있는 것들
end
CS:GetInstanceAddedSignal("Lava"):Connect(setupLava)   -- 나중에 생기는 것들

파트에 태그를 붙이는 법: Properties 창 하단 Tags 섹션에 Lava 입력. 맵 어디의 파트든 태그만 붙이면 자동으로 용암이 됩니다 — 맵 제작과 코드의 깔끔한 분업.

Attribute — 인스턴스에 변수 달기

-- Properties 하단 Attributes에서 추가하거나 코드로:
door:SetAttribute("RequiredKey", "GoldKey")
local key = door:GetAttribute("RequiredKey")
door:GetAttributeChangedSignal("Locked"):Connect(update)

태그 = "이건 무엇이다", Attribute = "이것의 설정값". 둘을 합치면 맵에서 값만 바꿔 동작이 달라지는 데이터 주도 설계가 됩니다 — 문마다 다른 열쇠, 점프대마다 다른 높이.

27애니메이션 재생 고급

만든 애니메이션을 코드로 재생하고, 타이밍에 맞춰 이벤트를 발사합니다.

  1. Animation Editor(Avatar 탭)로 애니메이션 제작 > Publish — Asset ID 획득. (블렌더에서 만들어 임포트도 가능)
  2. 재생:
local animation = Instance.new("Animation")
animation.AnimationId = "rbxassetid://123456789"

local animator = humanoid:FindFirstChildOfClass("Animator")
local track = animator:LoadAnimation(animation)

track:Play()
track:AdjustSpeed(1.5)            -- 1.5배속
track.Looped = false
track.Stopped:Wait()              -- 끝날 때까지

핵심 개념

  • Priority: Idle < Movement < Action < Action4. 공격 모션은 Action으로 — 걷기를 덮어쓰게.
  • 애니메이션 이벤트: 에디터에서 특정 프레임에 이벤트 마커 추가 > track:GetMarkerReachedSignal("Hit"):Connect(...) — 칼이 휘둘러지는 정확한 프레임에 판정/이펙트 발사. 액션 게임의 표준 기법.
  • 서버에서 Play하면 모두에게 복제 — 일반적으로 서버에서 재생합니다.

28디버깅 심화 고급

print 졸업. 중단점, 워치, 로그 전략으로 버그를 체계적으로 추적합니다.

중단점 (Breakpoint)

  1. 코드 줄 번호 왼쪽 클릭 — 빨간 점. 실행이 거기서 멈춥니다.
  2. 멈춘 상태에서 변수에 마우스를 올리면 현재 값이 보입니다. Watch 창에 식을 등록하면 계속 추적.
  3. F10 한 줄씩 / F11 함수 안으로 / F5 계속.

에러를 다루는 코드

local ok, err = pcall(function()
    riskyThing()
end)
if not ok then
    warn("실패:", err)       -- warn = 노란 출력, 죽지 않음
end

assert(player, "player가 nil입니다")   -- 조건 거짓이면 즉시 에러

고급 디버깅 수칙

  • 이분 탐색: 코드 절반을 주석 처리 — 버그가 남으면 남은 절반에, 사라지면 주석 친 절반에. 반복하면 몇 번 만에 범인 도달.
  • 재현 조건 고정: "가끔 발생"은 추적 불가. 항상 발생하는 최소 조건을 먼저 찾는 것이 일의 절반.
  • 최근 변경 의심: 어제까지 됐다면 범인은 오늘 바꾼 곳 — 버전 관리가 디버깅 도구인 이유.
  • Output에서 에러 줄 클릭 = 해당 코드로 점프. 호출 스택(Call Stack)으로 "누가 이 함수를 불렀나" 역추적.

29메모리와 연결 정리 고급

오래 켜둔 서버가 느려지는 이유 — 끊지 않은 연결과 지우지 않은 참조. 누수를 막는 습관.

누수의 전형

-- 나쁨: 캐릭터가 리스폰할 때마다 연결이 쌓임
player.CharacterAdded:Connect(function(char)
    RunService.Heartbeat:Connect(function() ... end)  -- 영원히 안 끊김!
end)
-- 좋음: 수명에 맞춰 정리
player.CharacterAdded:Connect(function(char)
    local conn = RunService.Heartbeat:Connect(function() ... end)
    char.AncestryChanged:Connect(function(_, parent)
        if not parent then conn:Disconnect() end  -- 캐릭터 제거 시 해제
    end)
end)

정리 규칙

  • 만든 연결은 수명이 끝날 때 Disconnect — "누가 언제 이 연결을 끊는가"를 만들 때 정합니다.
  • Destroy()는 인스턴스의 연결을 함께 끊어줍니다 — 인스턴스 기반이면 Destroy가 정리의 기본.
  • 테이블에 플레이어를 키로 저장했다면 PlayerRemoving에서 data[player] = nil — 안 지우면 나간 유저가 메모리에 영원히.
  • 연결과 인스턴스를 테이블에 모아뒀다 한 번에 정리하는 패턴(Maid/Janitor/Trove)이 커뮤니티 표준 — 초고급에서 구조화합니다.

30종합 프로젝트 — 상점 시스템 고급

고급 전체 조합: 모듈 + RemoteEvent + DataStore + 검증. 실제 게임의 축소판.

설계 먼저

ReplicatedStorage
  Remotes/BuyItem (RemoteEvent)
  ItemData (ModuleScript)        -- 아이템 정의 (공용: 클라가 가격 표시에도 씀)
ServerScriptService
  DataManager (ModuleScript)     -- 저장/로드 (챕터 23)
  ShopServer (Script)            -- 구매 검증 처리
-- ItemData (ModuleScript)
return {
    Sword  = {price = 100, type = "weapon"},
    Shield = {price = 150, type = "armor"},
}
-- ShopServer (Script) — 서버 검증이 핵심
local ItemData = require(RS.ItemData)
RS.Remotes.BuyItem.OnServerEvent:Connect(function(player, itemName)
    local item = ItemData[itemName]
    if not item then return end                       -- 1. 존재 검증
    if DataManager.getCoins(player) < item.price then return end  -- 2. 잔액
    if DataManager.hasItem(player, itemName) then return end      -- 3. 중복
    DataManager.addCoins(player, -item.price)
    DataManager.giveItem(player, itemName)
end)
  1. GUI(LocalScript)는 ItemData를 require해 목록과 가격을 그리고, 버튼 클릭 시 FireServer만.
  2. 도전 과제: 구매 결과를 FireClient로 돌려보내 성공/실패 알림 띄우기, 장착 시스템 추가.
TIP

"클라이언트는 그리기와 요청만, 서버는 검증과 변경만" — 이 한 줄이 고급의 전부입니다. 초고급에서는 이 구조를 게임 전체 규모로 키우는 아키텍처를 다룹니다.
