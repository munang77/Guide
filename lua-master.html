31게임 아키텍처 설계 초고급

스크립트 30개가 서로를 require하며 엉키기 전에 — 전체 구조를 먼저 그립니다.

서비스 패턴

ServerScriptService
  Main (Script)            -- 유일한 진입점: 서비스들을 순서대로 Init/Start
  Services/
    DataService            -- 저장/로드만
    CombatService          -- 전투 규칙만
    ShopService            -- 상점만
  Components/              -- 태그 기반 오브젝트 동작 (Lava, Door...)
  • 단일 진입점: Script는 Main 하나. 나머지는 전부 ModuleScript — 실행 순서를 Main이 통제합니다.
  • 2단계 초기화: 모든 서비스 Init()(자기 준비) 완료 후 Start()(서로 연결). 순환 의존 문제의 표준 해법.
  • 단방향 의존: 상위(게임 규칙)가 하위(유틸)를 require하고 역방향은 금지. 하위가 상위에 알릴 일은 시그널(이벤트)로.
  • Knit 같은 커뮤니티 프레임워크가 이 패턴의 기성품 — 직접 한 번 만들어 보면 프레임워크를 평가할 눈이 생깁니다.

상태 머신

-- 라운드 흐름: 명시적 상태 + 허용된 전이만
local state = "Lobby"
local transitions = {
    Lobby   = {"Playing"},
    Playing = {"Ending"},
    Ending  = {"Lobby"},
}
local function setState(new)
    assert(table.find(transitions[state], new), "잘못된 전이")
    state = new
    stateChanged:Fire(new)   -- 각 시스템이 구독
end

"게임이 가끔 이상한 상태에 빠져요"의 99%는 암묵적 상태(불리언 여러 개의 조합) 때문입니다. 상태를 하나의 명시적 값으로.

32타입 체킹 — --!strict 초고급

실행해야 아는 버그를 타이핑하는 순간 잡아내는 Luau의 강력한 무기.

--!strict
type ItemInfo = {
    name: string,
    price: number,
    stack: number?,        -- ?는 "없어도 됨(nil 허용)"
}

local function buy(player: Player, item: ItemInfo): boolean
    if getCoins(player) < item.price then
        return false
    end
    return true
end

buy(player, {name = "검"})   -- 빨간 줄! price가 없음 — 실행 전에 발견
  • 파일 첫 줄 --!strict — 이 파일의 타입을 엄격 검사. 에디터에 즉시 빨간 줄.
  • type으로 데이터 모양을 선언 — 문서이자 검사기. ItemData처럼 팀이 공유하는 구조에 특히 가치.
  • 유니온 "sword" | "bow", 제네릭 {T} 등 표현력 충분.
  • 처음엔 귀찮지만, "오타로 nil이 흘러들어 3시간 디버깅"을 0초로 만드는 투자입니다. 새 모듈부터 strict로 시작하세요.

타입 표기 빠른 참조

표기뜻예시
number string boolean기본 타입local n: number
T?nil 허용 (있어도/없어도)name: string?
{T}T의 배열{number} = 숫자 목록
{[K]: V}딕셔너리{[string]: number}
A | BA 또는 B (유니온)"on" | "off"
(A) -> B함수 타입(number) -> string
type 이름 = ...타입 별칭 정의type Vec = {x:number, y:number}
local x = y :: T타입 단언 (강제 지정)확실할 때만

strict 모드 3단계

  • --!nocheck — 타입 검사 끔 (옛 코드 임시 보호).
  • --!nonstrict — 기본값. 명백한 것만 검사.
  • --!strict — 엄격. 새 코드는 이걸로 시작 권장.
TIP

전체를 한 번에 strict로 바꾸면 빨간 줄 폭탄을 맞습니다. 새로 만드는 모듈부터 strict로 시작하고, 기존 코드는 건드릴 때마다 하나씩 올리세요.

33네트워킹 최적화 초고급

RemoteEvent를 마구 쏘면 렉이 됩니다. 보내는 양과 횟수를 설계합니다.

원칙

  • 빈도 제한: 매 프레임 FireServer 금지. 이동 같은 연속 데이터는 0.1초 묶음(스로틀)으로.
  • 배칭: 코인 10개 획득을 10번 쏘지 말고 1번에 {count = 10}.
  • 작게 보내기: 인스턴스 대신 이름/ID. 전체 상태 대신 바뀐 부분(델타)만.
  • 복제 활용: 서버가 바꾼 속성은 자동 복제 — 그걸 또 RemoteEvent로 알릴 필요 없음. Attribute 변경 + GetAttributeChangedSignal 구독이 공짜 동기화 채널.
  • 관심 영역: FireAllClients 남발 금지 — 가까운 플레이어에게만 FireClient (이펙트, 사운드).

서버 → 클라 이펙트 패턴

-- 서버: "무엇이 일어났다"는 사실만 가볍게 방송
RS.FX.PlayFX:FireAllClients("Explosion", position)

-- 클라: 무거운 연출(파티클/사운드/카메라 흔들림)은 각자 로컬에서
RS.FX.PlayFX.OnClientEvent:Connect(function(name, pos)
    FXLibrary[name](pos)
end)

이펙트를 서버에서 직접 생성하면 모든 파트가 복제로 전송됩니다 — 이름과 좌표만 보내고 클라가 만들게 하는 것이 수십 배 가볍습니다.

TIP

Studio > Network 진단(Shift+F3 Stats의 Recv/Send)으로 초당 송수신량 확인. 플레이어당 수신이 50KB/s를 넘보면 설계를 의심하세요.

34보안과 안티 익스플로잇 초고급

익스플로이터는 클라이언트의 모든 것을 보고, 바꾸고, 아무 Remote나 아무 값으로 쏠 수 있습니다. 그 전제로 설계합니다.

위협 모델

  • 클라이언트 메모리/스크립트 — 읽기·조작 가능. LocalScript의 검사 로직은 장식입니다.
  • RemoteEvent — 누구나, 어떤 인자로든, 초당 수백 번 호출 가능.
  • 자기 캐릭터 — 속도/순간이동 조작 가능 (물리는 클라이언트 권한).

서버 검증 체크리스트 (모든 Remote마다)

  1. 타입 검증: if typeof(itemName) ~= "string" then return end — 테이블 폭탄, nil 공격 차단.
  2. 존재/권한 검증: 그 아이템이 실제로 있나, 이 플레이어가 할 수 있는 상태인가.
  3. 가능성 검증: 거리(상점 옆에 있나), 쿨다운(0.2초 전에 또 공격?), 수치 범위(데미지를 클라가 보냈다면 설계 오류 — 서버가 계산해야 함).
  4. 빈도 제한: 플레이어별 호출 횟수 카운트, 초과 시 무시 + 기록.
-- 이동 핵 완화: 서버에서 주기적 순간이동 검사
local last = {}
RunService.Heartbeat:Connect(function()
    for _, plr in Players:GetPlayers() do
        local root = plr.Character and plr.Character.PrimaryPart
        if root then
            local prev = last[plr]
            if prev and (root.Position - prev).Magnitude > 60 then
                root.CFrame = CFrame.new(prev)   -- 되돌림 (킥보다 안전)
            end
            last[plr] = root.Position
        end
    end
end)
!!

완벽한 방어는 없습니다. 목표는 "치팅의 이득을 서버가 인정하지 않게" — 돈, 아이템, 킬 판정이 전부 서버 계산이면 클라 조작은 자기 화면 낙서로 끝납니다.

35성능 프로파일링 초고급

"느린 것 같다"가 아니라 측정해서 어디가 몇 ms인지 찾아내는 기술.

도구

  • MicroProfiler (Ctrl+F6): 프레임 타임라인. 자기 코드를 구간으로 표시하려면:
debug.profilebegin("EnemyUpdate")
updateAllEnemies()
debug.profileend()   -- MicroProfiler에 EnemyUpdate 막대로 표시됨
  • ScriptProfiler (Studio 콘솔): 어느 함수가 CPU를 먹는지 순위표.
  • os.clock() 수동 측정: 의심 구간 앞뒤로 — 가장 원시적이고 가장 확실.

흔한 병목과 처방

병목증상처방
매 프레임 전체 순회적 수에 비례해 프레임 하락갱신 분산(프레임당 1/N씩), 멀면 갱신 주기 낮추기
GetDescendants 반복 호출Heartbeat 스파이크결과 캐시, 태그로 대체
문자열 연결 누적 (..)GC 스파이크table.concat, string.format
Instance 대량 생성/파괴총알/이펙트 때 끊김오브젝트 풀링 — 만들지 말고 재사용
 
-- 오브젝트 풀 핵심
local pool = {}
local function get()
    local b = table.remove(pool) or template:Clone()
    b.Parent = workspace
    return b
end
local function release(b)
    b.Parent = nil            -- Destroy 대신 회수
    table.insert(pool, b)
end

36병렬 Luau — Actor 초고급

무거운 계산을 다른 CPU 코어로. 단, 아무거나 병렬이 되는 건 아닙니다.

구조

  1. Actor 인스턴스 안에 스크립트를 넣으면 그 스크립트는 독립 실행 단위가 됩니다.
  2. task.desynchronize() — 여기서부터 병렬 구간 (다른 코어에서 실행 가능).
  3. 병렬 구간에서는 읽기만 안전. 인스턴스 속성 쓰기는 task.synchronize()로 직렬 구간에 돌아와서.
  4. Actor 간 통신: actor:SendMessage() / BindToMessageParallel.
-- 패턴: 병렬로 계산 → 직렬로 적용
task.desynchronize()
local results = heavyPathfindingMath(data)   -- 계산 (코어 분산)
task.synchronize()
applyResults(results)                         -- 쓰기 (메인)

언제 쓰나

  • 적합: 다수 유닛의 경로/AI 계산, 지형 생성 수학, 대량 Raycast — 계산량이 크고 결과만 합치면 되는 일.
  • 부적합: 평범한 게임 로직 — 동기화 비용이 이득을 잡아먹습니다. 프로파일링으로 병목 확인 후 마지막 수단으로.
  • 벤치마크 필수: Actor 분할 자체에 비용이 있어 작은 작업은 오히려 느려집니다.

37메모리 추적과 GC 초고급

12시간 켜진 서버가 기어가는 이유를 찾아내는 법.

측정

  • 개발자 콘솔(F9) > Memory 탭 — 카테고리별(LuaHeap, Instances, ...) 사용량. 시간에 따라 우상향하는 카테고리가 누수.
  • gcinfo() — Lua 힙 KB. 주기적으로 로그를 남겨 추세 감시.
  • 의심 시 부하 테스트: 입장/퇴장을 수십 번 반복 — 퇴장 후에도 메모리가 안 돌아오면 플레이어 참조 누수.

Luau GC 이해

  • 증분(incremental) GC — 조금씩 자동 수거. 참조가 하나라도 남으면 영원히 수거 안 됨.
  • 누수 3대장: ① 끊지 않은 이벤트 연결(클로저가 잡은 모든 것이 생존) ② 테이블에 남은 플레이어/캐릭터 참조 ③ Parent=nil로 떼고 Destroy 안 한 인스턴스.
  • 캐시에는 약한 테이블: setmetatable({}, {__mode = "k"}) — 키가 다른 곳에서 사라지면 자동 제거. 플레이어 키 캐시에 적합.
-- 수명 객체 정리 패턴 (Trove/Maid 직접 구현 골격)
local Trove = {}
Trove.__index = Trove
function Trove.new() return setmetatable({items = {}}, Trove) end
function Trove:add(item) table.insert(self.items, item) return item end
function Trove:clean()
    for _, it in self.items do
        if typeof(it) == "RBXScriptConnection" then it:Disconnect()
        elseif typeof(it) == "Instance" then it:Destroy() end
    end
    table.clear(self.items)
end

38buffer와 직렬화 초고급

데이터를 바이트 수준으로 다루는 도구. 네트워크와 저장 용량을 극한으로 줄입니다.

local b = buffer.create(9)            -- 9바이트
buffer.writeu8(b, 0, 3)               -- [0]: 아이템 종류 (0~255)
buffer.writef32(b, 1, pos.X)          -- [1-4]: X 좌표
buffer.writef32(b, 5, pos.Z)          -- [5-8]: Z 좌표

RS.Spawn:FireAllClients(b)            -- 9바이트 전송

-- 수신 측
local kind = buffer.readu8(b, 0)
local x = buffer.readf32(b, 1)
  • 같은 정보를 테이블 {kind=3, x=12.5, z=-40}로 보내면 수십 바이트 — buffer면 9바이트. 초당 수백 건 이벤트에서 차이가 큽니다.
  • 타입 선택: u8(0~255), i16, u32, f32(소수), f64. 좌표는 f32면 충분합니다.
  • DataStore 절약에도: 대량 데이터를 buffer로 패킹 → base64 문자열로 저장하면 JSON 대비 수 배 압축.
  • 대가는 가독성: 오프셋 명세를 주석/타입으로 문서화하고, 버전 바이트를 맨 앞에 — 포맷이 바뀌어도 옛 데이터를 읽게.
  • 적용 기준: 프로파일링으로 네트워크/저장이 병목임을 확인한 곳에만. 평소엔 테이블이 옳습니다.

39테스트 자동화 초고급

고치면 다른 데가 터지는 공포를 끝내는 법 — 컴퓨터가 대신 검사하게.

테스트 가능한 코드

  • 전제: 로직을 인스턴스에서 분리. calcDamage(att, def)처럼 값을 받아 값을 돌려주는 순수 함수는 테스트가 한 줄, Humanoid를 직접 만지는 함수는 테스트 불가.
  • 구조: 규칙(순수 모듈) ↔ 적용(인스턴스 조작) 분리 — 아키텍처(챕터 31)와 같은 방향.
-- 미니 테스트 러너 (직접 만들어도 이 정도면 충분)
local function test(name, fn)
    local ok, err = pcall(fn)
    print((ok and "[통과] " or "[실패] ") .. name, err or "")
end
local function expectEqual(a, b)
    if a ~= b then error(tostring(a) .. " ~= " .. tostring(b), 2) end
end

test("크리티컬은 2배", function()
    expectEqual(Damage.calc(10, {crit = true}), 20)
end)
test("데미지는 0 미만 불가", function()
    expectEqual(Damage.calc(-5, {}), 0)
end)
  • 테스트 스크립트를 ServerScriptService/Tests에 두고 개발 중에만 활성화. 본격적으로는 Jest-Lua/TestEZ 같은 프레임워크.
  • 버그를 고칠 때마다 그 버그의 테스트를 추가 — 같은 버그의 재발이 0이 됩니다.
  • 경계값부터: 0, 음수, 빈 테이블, nil — 버그는 항상 가장자리에 삽니다.

40외부 도구 체인 — Rojo와 Wally 초고급

Studio 밖에서 코딩하는 프로 워크플로 — 진짜 에디터, 진짜 버전관리, 패키지 매니저.

Rojo — 파일 시스템 ↔ Studio 동기화

  1. 코드를 .luau 파일로 폴더에 작성 (VS Code + Luau LSP 확장 — 강력한 자동완성과 strict 검사).
  2. Rojo가 폴더 구조를 Studio 트리에 실시간 동기화 — 저장하면 Studio에 즉시 반영.
  3. 이제 코드가 텍스트 파일이므로: Git으로 버전관리, 브랜치, 코드 리뷰(PR), 협업 충돌 해결이 전부 가능해집니다.

Wally — 패키지 매니저

# wally.toml — 커뮤니티 라이브러리를 선언적으로 설치
[dependencies]
Trove = "sleitnick/trove@1"
Promise = "evaera/promise@4"
  • 검증된 라이브러리(정리 도구, Promise, 네트워킹 래퍼)를 버전 고정으로 가져옴 — 직접 복붙해 관리하는 것보다 안전.
  • 가져온 코드도 읽어보는 습관 — 초고급의 라이브러리 사용은 "이해하고 위임"입니다.
TIP

혼자 + 소규모면 Studio만으로 충분합니다. 도구 체인은 팀 협업, 장기 프로젝트, 코드 리뷰가 필요해지는 시점의 투자입니다. 전문가 단계에서는 이 위에 라이브 게임 운영을 올립니다.
