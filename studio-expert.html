Studio 전문가 — 상업 규모의 맵 제작

아트 디렉션, 성능 예산, 초대형 월드, 팀 협업, 라이브 운영, 외주. 맵 제작을 직업 수준으로 — 진짜 전문가의 영역.

10챕터 아트 디렉션 팀 협업 라이브 운영

41아트 디렉션 전문가

"예쁘게"가 아니라 "이 게임답게" — 모든 시각 결정의 기준이 되는 규칙서를 만드는 일.

아트 바이블 만들기

  1. 핵심 형용사 3개: 게임의 느낌을 3단어로 (예: "아늑한, 손으로 만든, 동화 같은"). 모든 결정을 이 3단어로 검사합니다.
  2. 형태 언어: 둥근가 각진가, 과장인가 사실인가, 디테일 밀도는 — 소품 하나 추가할 때마다의 기준.
  3. 팔레트와 금지색 (초고급 35의 확장), 머티리얼 규칙(러프니스 범위, 채도 한계), 스케일 과장 규칙(문은 실제보다 1.3배 등).
  4. 레퍼런스 시트: 잘된 예 / 안 되는 예를 나란히 — 글보다 그림이 규칙을 전달합니다.

디렉션 실무

  • 일관성 > 개별 퀄리티: 뛰어난 사실적 바위 하나가 카툰 맵 전체를 깨뜨립니다 — "잘 만들었지만 우리 게임이 아님"을 거절하는 것이 디렉터의 일.
  • 벤치마크 숏: 맵에서 가장 아름다워야 할 장면 3곳을 먼저 완성도 100%로 — 나머지 전체의 기준점이 됩니다.
  • 주기적 스타일 감사: 한 달에 한 번 맵 전체를 둘러보며 바이블 위반을 수집 — 일관성은 유지보수 대상입니다.

아트 바이블에 들어갈 항목

팀의 모든 시각 결정 기준이 되는 문서입니다. 이 표를 채우면 한 장짜리 바이블이 완성됩니다.

항목정할 내용예시
핵심 형용사 3개게임의 느낌아늑한 / 손으로 만든 / 동화 같은
형태 언어둥근가 각진가, 과장 정도둥글고 과장된 카툰
팔레트주조색 + 보조색 + 강조색따뜻한 파스텔 + 청록 강조
금지색쓰지 않을 색채도 높은 형광은 보스만
머티리얼 규칙러프니스·반사 범위매트 위주, 금속은 포인트만
스케일 과장실제 대비 비율문·창문 1.3배 크게
조명 톤구역별 색온도 규칙안전=따뜻, 위험=차가움
레퍼런스 시트좋은 예 / 나쁜 예이미지로 나란히
TIP

바이블이 추상적이면 안 지켜집니다. "따뜻하게"가 아니라 "Ambient (120,110,90), 그림자에 파랑 5% 섞기"처럼 수치로 적어야 팀이 똑같이 재현합니다.

42퍼포먼스 버짓 설계 전문가

"느려지면 최적화"는 아마추어 — 전문가는 시작 전에 예산을 정하고 그 안에서 만듭니다.

예산표 만들기

항목예산 예시 (모바일 기준)측정
화면 내 파트 수≤ 5,000Stats(Shift+F3) > Render
화면 내 트라이앵글≤ 150만MicroProfiler / Stats
그림자 광원구역당 ≤ 3수동 카운트
활성 파티클≤ 600Rate × Lifetime 합산
텍스처 메모리≤ 300MBF9 콘솔 > Memory
인스턴스 총수≤ 30만F9 > Instances

운용 규칙

  • 예산은 구역별로 배분: 허브(사람 몰림)는 빡빡하게, 외곽은 여유 있게 — 총합이 아니라 최악 지점이 예산입니다.
  • 최저 사양 기기를 정하고 그 기기에서 측정 — "내 PC에선 잘 돌아요"는 데이터가 아닙니다.
  • 초과 시 트레이드 절차: 새 디테일을 넣으려면 무엇을 뺄지 함께 제출 — 예산은 협상이 아니라 회계.
  • 주간 측정 기록: 같은 지점 3곳의 프레임/메모리를 매주 기록 — 그래프가 우하향하기 시작하는 시점을 즉시 포착.
TIP

예산의 80%만 쓰고 20%는 비워 두세요 — 시즌 장식(초고급 39)과 라이브 이벤트가 그 여유를 씁니다.

43초대형 월드 — 멀티 플레이스 전문가

한 플레이스의 한계를 넘는 규모 — 월드를 여러 플레이스로 나누고 텔레포트로 잇습니다.

분할 설계

  1. 유니버스 구조: 시작 플레이스(허브/로비) + 구역 플레이스들(대륙, 던전, 이벤트 맵) — 한 게임(Experience) 아래 여러 플레이스.
  2. 분할 기준: 테마가 완전히 다른 곳, 동시에 갈 일 없는 곳, 메모리가 함께 못 사는 곳 — 한 지도 안에서 걸어가야 자연스러운 곳은 분할 금지 (StreamingEnabled로 해결).
  3. 경계 연출: 텔레포트 지점을 서사적 관문(배, 포탈, 엘리베이터)으로 — 로딩을 세계관으로 감쌉니다.

이동 설계 (맵 제작자의 관점)

  • 출발 게이트와 도착 지점을 1:1로 짝 — 도착 지점은 안전지대(스폰 킬 방지)이고 돌아가는 게이트가 보이는 곳에.
  • 로딩 화면용 비주얼: 플레이스마다 대표 일러스트/스크린샷을 준비 — 이동 중 보이는 화면도 맵의 일부.
  • 각 플레이스에 미니 허브: 도착 광장에 지도/안내판 — 처음 온 플레이어의 방향감각을 30초 안에.
  • 구역 간 시각 연속성: 허브에서 보이던 산이 산 플레이스의 스카이박스에도 — 세계가 하나라는 감각 유지.

플레이스 간 데이터(인벤토리 등)는 스크립트 영역이므로 Luau 강좌(고급 23, 전문가 44)와 만나는 지점입니다 — 맵 제작자는 "어디서 넘어가고 어디로 도착하는가"의 경험을 책임집니다.

44에셋 파이프라인 자동화 전문가

에셋 수백 개를 손으로 임포트하는 팀은 망합니다 — 흐름을 자동화 가능한 규격으로.

표준 흐름

  1. 제작 규격서: 블렌더 전문가(49)의 납품 스펙을 Studio 쪽에서 발행 — 단위, 피벗, 트라이 예산, 텍스처 채널 규칙, 네이밍(SM_구역_이름_번호).
  2. 일괄 임포트: 3D Importer로 여러 메시 한 번에, Asset Manager로 텍스처 벌크 업로드 — 한 개씩 끌어넣기 금지.
  3. 임포트 후 검사 체크리스트: 스케일(기준 캐릭터 옆에 배치), 충돌 모드, 머티리얼 연결, Anchored — 검사 안 된 에셋은 라이브러리에 못 들어가는 규칙.
  4. 맵 라이브러리 플레이스: 검수 통과 에셋만 모아둔 전시장 플레이스 — 제작자는 여기서 패키지로 가져다 씁니다.

버전과 추적

  • 에셋 ID 대장(스프레드시트): 이름 / ID / 버전 / 제작자 / 사용 구역 — "이 바위 어디서 왔지"를 끝냅니다.
  • 패키지 버전 잠금: 라이브 맵의 패키지는 자동 업데이트 끄고 검수 후 수동 갱신 — 라이브에서 갑자기 바뀌는 사고 방지.
  • 대량/반복 작업(수백 개 리네이밍, 일괄 속성 변경)은 Open Cloud API나 커맨드 바 자동화의 영역 — 도구화가 필요해지는 시점이 곧 전문가의 시점입니다.

45Team Create 협업 전문가

한 맵을 여럿이 동시에 — 충돌 없이, 서로를 지우지 않고 작업하는 규칙.

협업 규칙 (문서로 박제)

  1. 구역 소유제: 사람마다 담당 구역(폴더)을 — 같은 폴더를 두 명이 동시에 만지지 않습니다. Explorer 최상위 폴더가 곧 업무 분장표.
  2. 공용물 잠금: 라이팅 설정, 공용 키트, 스폰 — 변경은 지정 담당자만, 변경 시 팀 공지.
  3. 네이밍 강제: "Part(243개)"가 보이면 그날 작업 중지하고 정리 — 협업에서 무명 파트는 남의 시간 도둑.
  4. 일일 정리 시간: 세션 끝에 10분 — 임시 파트 삭제, 폴더 귀속, 이름 부여.

안전장치

  • 버전 히스토리(File > Game Settings에서 확인 가능한 Version History)로 복원 지점 확보 — 큰 작업 전 퍼블리시로 스냅샷.
  • 실험은 별도 플레이스에서: 본 맵 복사본에서 시도 → 확정되면 패키지/모델로 본 맵에 이식.
  • 드래프트 모드(Collaborative Editing의 스크립트 드래프트)와 별개로, 빌드 작업은 실시간 반영임을 전원이 인지 — "잠깐 테스트로 지워봄"이 라이브로 갑니다.
  • 리뷰 워크플로: 구역 완성 시 담당자가 아트 디렉터(챕터 41)에게 워크스루 — 둘러보며 바이블 위반/예산 초과 점검.

46접근성과 UX 전문가

더 많은 사람이 편하게 — 접근성은 배려이자, 플레이어 수를 직접 늘리는 설계입니다.

시각 접근성

  • 색에만 의존 금지: 빨강/초록 구분(색약 인구 약 8%)이 게임플레이 정보라면 모양/패턴/위치를 병행 — 적 팀은 빨강이면서 뾰족하게.
  • 명도 대비 우선: 흑백 테스트(초고급 35)를 통과하면 대부분의 색각에서 안전.
  • 중요 정보의 크기: 모바일 화면에서 표지판/아이콘이 읽히는가 — 고급(29)의 모바일 점검을 접근성 기준으로 격상.
  • 광과민 배려: 강한 점멸(초당 3회 이상 풀스크린 플래시)은 사용하지 않습니다.

길찾기 UX

  • 3중 안내: 랜드마크(멀리) + 리딩 라인(중간) + 표지판(가까이) — 한 채널을 놓쳐도 다른 채널이 받칩니다.
  • 막다른 길엔 보상: 헛걸음이 아니라 "탐험 성공"으로 기억되게.
  • 실수 용서: 낙사 구간엔 복귀 경로, 한 번의 미스가 5분 손해로 이어지지 않게 — 난이도와 불친절은 다릅니다.
  • 첫 30초 설계: 스폰 지점에서 보이는 것만으로 "여긴 어디고 뭘 하면 되는지"가 전달되는가 — 신규 유저 이탈의 최대 변수.

47데이터로 맵 개선하기 전문가

초고급(40)의 관찰 테스트를 수치화 — 수천 명의 발자국이 말해주는 것을 읽습니다.

맵 지표 설계

  • 히트맵: 플레이어 위치를 주기적으로 기록(스크립트 팀과 협업) — 몰리는 곳/아무도 안 가는 곳이 색으로 드러납니다.
  • 구역 체류 시간: 의도한 페이싱(초고급 37)과 실제가 일치하는가 — 숨 고르기 구역을 다들 뛰어 지나가면 설계 실패.
  • 사망/이탈 지점: 같은 낙사 지점에서 수백 명이 죽으면 그건 난이도가 아니라 결함.
  • 도달률 퍼널: 스폰 → 구역1 → 구역2 ... 어디서 몇 %가 증발하는가.

가설 → 변경 → 검증

  1. 데이터에서 이상 발견: "동쪽 시장에 5%만 방문".
  2. 가설 수립: 입구가 어두워서 / 보상이 없어서 / 존재를 몰라서.
  3. 가장 싼 변경 먼저: 입구 조명 추가 (1시간) → 재측정 → 부족하면 다음 가설.
  4. A/B가 가능하면(전문가 Luau 46) 두 버전을 절반씩 — 확신이 아니라 비교로 결정.
!!

데이터는 "무엇이 일어나는가"만 말합니다. "왜"는 여전히 관찰 테스트와 디자이너의 해석 — 숫자가 시키는 대로만 고치면 개성 없는 평균의 맵이 됩니다.

48라이브 맵 운영 전문가

유저가 매일 들어오는 맵을 고치는 법 — 수술은 환자가 깨어 있는 채로 진행됩니다.

변경 관리

  1. 스테이징 플레이스: 모든 변경은 복사본에서 작업 + 검증 → 본 플레이스에 이식 → 퍼블리시. 라이브 직접 편집은 긴급 수리뿐.
  2. 퍼블리시 = 배포 의식: 체크리스트(예산 측정, 스폰 확인, 충돌 점검, 모바일 확인) 통과 후, 트래픽 낮은 시간대에.
  3. 롤백 준비: 퍼블리시 직전 버전 번호를 기록 — 문제 시 Version History에서 즉시 복원. 복원 절차를 평시에 연습해 두세요.
  4. 변경 로그 공개: 유저 커뮤니티에 "무엇이 바뀌었나" — 신뢰는 운영의 자산입니다.

맵의 수명 관리

  • 콘텐츠 리듬: 큰 구역 추가(분기) + 작은 변화(월간: 장식, 비밀, 시즌) — 맵이 살아있다는 신호가 복귀 이유가 됩니다.
  • 구역 리노베이션: 데이터(챕터 47)로 죽은 구역을 찾아 재설계 — 추가만 하면 맵은 비대해지고 평균 품질이 떨어집니다.
  • 구버전 보존: 큰 개편 전 옛 맵을 박물관 플레이스로 — 고참 유저의 추억은 커뮤니티 자산.
  • 긴급 대응 키트: 문제 구역을 즉시 봉쇄할 차단벽 모델, 우회 경로 — 사고 때 만들기 시작하면 늦습니다.

49외주와 커미션 전문가

맵 제작이 수입이 되는 단계 — 견적, 계약, 납품의 프로 매너.

견적 산정

  1. 요구를 수치로: 맵 크기(스터드), 구역 수, 디테일 밀도(레퍼런스 스크린샷 기준), 지형 비중, 에셋 신규 제작 여부.
  2. 시간 견적 × 시급 + 예비 20% — "감으로 통짜 가격"은 둘 중 하나가 손해 봅니다.
  3. 마일스톤 분할: 그레이박스(30%) → 1차 아트패스(40%) → 폴리시+납품(30%) — 단계별 검수와 부분 정산.
  4. 수정 정책 명시: 마일스톤당 수정 2회, 이후 시간당 과금 — 분쟁의 90%는 수정 범위에서 납니다.

신뢰와 안전

  • 서면 합의: 범위, 금액, 기한, 수정, 저작권 귀속(보통 잔금 완납 시 이전), 포트폴리오 게재 허용 여부.
  • 중간 산출물은 워터마크/별도 플레이스 시연으로 — 잔금 전 전체 파일 전달은 위험.
  • 플랫폼 규칙 준수: 거래는 추적 가능한 수단으로, 미성년자 거래 보호 규정 확인.
  • 평판 = 다음 일감: 기한을 못 지킬 것 같으면 미리 알리는 것이 프로 — 잠수가 경력을 끝냅니다.

50포트폴리오와 팀 합류 전문가

전 과정의 끝 — 실력을 증명하고 더 큰 프로젝트로 나아가는 법.

맵 제작자의 포트폴리오

  • 플레이 가능한 쇼케이스: 스크린샷보다 강한 것은 직접 걸어볼 수 있는 맵 링크 — 동선, 페이싱, 사운드까지 전달됩니다.
  • 구성: 대표작 풀샷 → 무드별 구역 샷 → 그레이박스/완성 비교 → 예산 수치(파트 수, 프레임) — 과정과 규율을 함께 증명.
  • 역할 명시: 협업작은 내가 만든 부분을 정확히 — 과장은 면접 한 번에 무너집니다.
  • 영상 워크스루 60~90초: 조명이 좋은 시간대로, 카메라는 플레이어 시점 위주.

팀에 들어가기 / 꾸리기

  1. 지원 시 그 게임의 아트 스타일로 만든 팬 구역 하나 — "우리 게임을 이해한다"는 최고의 증명.
  2. 테스트 과제는 범위를 먼저 합의(무급 과제의 적정선은 수 시간 분량) — 범위 없는 무급 노동은 거절이 프로.
  3. 팀을 꾸린다면: 아트 바이블(41)과 협업 규칙(45)을 첫 주에 — 규칙 없는 재능 집단은 흩어집니다.
  4. 계속 성장: 다른 장르의 맵 모작, GDC 레벨 디자인 강연, 건축/조경/영화 미술 — 영감의 원천을 Roblox 밖에.
