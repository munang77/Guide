41스튜디오 파이프라인 설계 전문가

혼자가 아니라 팀과 시간을 상대로 작업하는 단계. 파일 하나가 아니라 시스템을 설계합니다.

프로젝트 구조 표준화

# 프로덕션 프로젝트 구조 예시
project/
  assets/        # 캐릭터, 소품별 폴더. 각각 model/ texture/ export/
  scenes/        # 작업 씬. 날짜나 버전 접미사 금지 — 버전관리가 할 일
  lib/           # 링크용 공용 라이브러리 (.blend)
  export/        # FBX/GLB 산출물. 자동 생성, 손으로 안 건드림
  scripts/       # 파이프라인 자동화 파이썬

네이밍 컨벤션

  • 오브젝트: SM_Rock_Large_01 (SM=Static Mesh), SK_Hero (SK=Skeletal). 접두사로 종류를 즉시 구분.
  • 재질: M_Rock, 텍스처: T_Rock_BC(BaseColor) / _N(Normal) / _R(Roughness).
  • 본(Bone): 좌우 대칭은 반드시 .L / .R 접미사 — 미러 기능들의 전제 조건.
  • 규칙은 문서로 박제하고 예외를 만들지 않는 것이 핵심. 자동화 스크립트는 규칙 위에서만 동작합니다.

버전 관리

  • .blend는 바이너리라 Git 기본만으론 비효율 — Git LFS로 대용량을 분리 추적.
  • 커밋 단위는 "작업 의미" 단위 (모델 완성, 리토폴로지 완료). 하루 끝 통커밋 금지.
  • 텍스처 원본(PSD/KRA)과 베이크 산출물(PNG)을 분리 — 산출물은 재생성 가능하므로 LFS 부담을 줄일 수 있음.
TIP

파이프라인의 성공 기준은 "새 멤버가 문서만 읽고 반나절 안에 에셋 하나를 규격대로 납품할 수 있는가" 입니다.

42대규모 씬 관리 전문가

오브젝트 수천 개, 수억 폴리곤의 씬을 멈추지 않고 다루는 기술.

Collection 전략

  • 씬을 Collection 트리로 구획: ENV(환경) / CHAR / PROPS / LIGHTS / FX. 뷰 레이어에서 통째로 켜고 끄며 작업 영역만 로드.
  • Collection Instance(Shift+A > Collection Instance)로 반복 배치 — 데이터는 1벌, 메모리 절약.

Link와 Library Override

  1. File > Link로 외부 .blend의 Collection을 링크 — 원본 수정이 모든 씬에 전파됩니다.
  2. 링크된 에셋을 씬별로 변형해야 하면 Library Override (우클릭 > Library Override > Make) — 원본은 그대로, 이 씬에서만 위치/머티리얼 덮어쓰기.
  3. 캐릭터 리그를 링크+오버라이드로 쓰는 것이 애니메이션 프로덕션의 표준입니다.

무거운 씬 생존 설정

기법방법효과
SimplifyRender Properties > Simplify, 뷰포트 Subdivision 0~1뷰포트 즉시 경량화
인스턴싱Alt+D 복제(Linked) 또는 Geometry Nodes Instance메시 데이터 공유
Bounds 표시Object Properties > Viewport Display > Bounds배경 덩어리를 상자로만 표시
Hide in Viewport모니터 아이콘 (렌더에는 포함)작업 중 안 보이게

43커스텀 노드 그룹과 에셋 배포 전문가

반복되는 노드 조합을 도구로 포장해 팀 전체가 쓰게 만듭니다.

노드 그룹 설계 원칙

  1. 노드 선택 > Ctrl+G로 그룹화. 입출력 소켓에 의미 있는 이름과 기본값, 최소/최대 범위를 지정.
  2. 내부 구현은 숨기고 소켓만 노출 — 쓰는 사람이 내부를 몰라도 되게. 함수를 설계하듯.
  3. 그룹 안에 그룹: 저수준 유틸(마스크 생성기) 위에 고수준 그룹(바위 재질)을 조립.
  4. Description 필드 작성 — 마우스 오버 툴팁이 곧 문서.

Asset Browser로 배포

  1. 완성된 노드 그룹/재질/오브젝트 우클릭 > Mark as Asset.
  2. Preferences > File Paths > Asset Libraries에 팀 공유 폴더(NAS, 클라우드 동기화 폴더) 등록.
  3. 카탈로그(Catalog)로 분류 트리 구성 — Materials/Nature, NodeGroups/Masks 처럼.
  4. 라이브러리 .blend는 "읽기 전용 + 담당자만 갱신" 규칙으로 — 아무나 덮어쓰면 전원이 깨집니다.
!!

노드 그룹을 수정하면 그걸 쓰는 모든 씬에 영향이 갑니다. 동작이 바뀌는 수정은 새 버전(v2 그룹)으로 추가하고 기존 것은 유지하는 것이 안전합니다 — 코드의 하위 호환성과 같은 개념.

44시뮬레이션 캐시 운용 전문가

시뮬레이션은 계산이 아니라 자산입니다. 캐시를 굽고, 보관하고, 재사용합니다.

캐시 전략

  • 모든 시뮬(천, 유체, 입자)은 Bake해서 디스크 캐시로 고정 — 재생할 때마다 재계산하지 않게.
  • 캐시 경로는 프로젝트 상대 경로(//cache/)로 — 다른 컴퓨터에서도 깨지지 않게.
  • 유체는 해상도별 캐시를 따로: 프리뷰(64)로 타이밍을 확정한 뒤 파이널(256+)을 밤새 베이크.
  • 대형 캐시는 버전관리에서 제외(.gitignore)하고 재생성 스크립트를 커밋.

Alembic으로 시뮬 굳히기

  1. 완성된 시뮬을 File > Export > Alembic(.abc)으로 베이크된 메시 시퀀스로 내보내기.
  2. 새 씬에서 .abc를 임포트하면 시뮬 설정 없이 결과만 재생 — 라이팅/렌더 씬을 가볍게 유지.
  3. 시뮬 담당과 라이팅 담당이 분업하는 표준 방식입니다 (챕터 36 USD/Alembic 참고).

결정론과 재현성

같은 씬을 다시 열어도 같은 결과가 나오게 — Seed 값 기록, 프레임 시작점 고정, 의존 오브젝트의 애니메이션을 먼저 베이크. "어제는 됐는데 오늘 다르게 나온다"는 대부분 캐시가 아닌 라이브 시뮬을 렌더에 태웠기 때문입니다.

45애드온 아키텍처와 배포 전문가

단일 파일 스크립트(챕터 40)를 넘어 — 멀티 모듈, 설정 저장, 업데이트까지 갖춘 진짜 애드온.

멀티파일 구조

# 패키지형 애드온 구조
my_addon/
  __init__.py    # bl_info, register/unregister만. 로직 금지
  operators.py   # Operator 클래스들
  panels.py      # UI Panel
  props.py       # PropertyGroup (설정 데이터)
  core/          # 블렌더 비의존 순수 로직 — 단위 테스트 가능
  utils.py
  • core 분리가 핵심: bpy를 import하지 않는 순수 파이썬 로직을 분리하면 블렌더 없이 테스트할 수 있습니다.
  • 클래스 등록은 리스트로 모아 일괄 register — 등록 순서 버그를 줄입니다.
  • 설정은 AddonPreferences에 저장 — 블렌더를 꺼도 유지되는 사용자 설정.

배포와 호환성

  • 블렌더 4.2+는 Extensions 플랫폼(extensions.blender.org) — manifest(blender_manifest.toml) 작성, 심사 후 공개 배포.
  • 버전 분기: if bpy.app.version >= (4, 1, 0): 로 API 변경 대응. 지원 최소 버전을 bl_info에 명시.
  • 에러 리포트 받을 채널(GitHub Issues)과 변경 로그를 갖추는 순간부터 "도구"가 "제품"이 됩니다.

46Geometry Nodes 시스템 설계 전문가

노드 몇 개가 아니라, 맵 하나를 통째로 생성하는 프로시저럴 시스템을 설계합니다.

시스템 분해

예: "숲 생성기"를 한 트리로 만들지 말고 책임별로 분해합니다 —

  1. 분포 레이어: 어디에 심을지 (경사도/고도/노이즈 마스크 조합 → 포인트).
  2. 선택 레이어: 무엇을 심을지 (포인트마다 가중 랜덤으로 나무 종류 선택).
  3. 변형 레이어: 어떻게 심을지 (크기/회전/기울기 랜덤, 지형 노멀 정렬).
  4. 각 레이어를 노드 그룹으로 — 분포만 갈아끼우면 바위 생성기로 재사용됩니다.

성능 규율

  • Realize Instances는 최후에: 인스턴스 상태로 끝까지 끌고 가면 수십만 그루도 가볍습니다. Realize하는 순간 메모리가 터집니다.
  • 스파이크 찾기: 노드 에디터 헤더 > Timings 표시 — 어느 노드가 ms를 먹는지 보입니다.
  • Named Attribute로 레이어 사이 데이터 전달 — 스파게티 연결 대신 명세된 인터페이스.
TIP

입력 슬라이더는 "예술가 언어"로: Density 0.73 대신 "나무 빽빽함 0~10". 도구의 품질은 내부가 아니라 노출된 입력의 품질입니다.

47셰이더 최적화와 디버깅 전문가

느린 재질, 노이즈 끼는 재질, 게임에서 무거운 재질을 진단하고 수술합니다.

비용 진단

  • 렌더 시간이 튀면 재질을 하나씩 Holdout으로 바꿔 범인 격리 — 대부분 SSS, 볼륨, 다층 투명이 범인.
  • 텍스처 해상도 감사: 화면에서 차지하는 픽셀보다 큰 텍스처는 낭비. 소품은 1K면 충분한 경우가 대부분.
  • 노드 수가 아니라 텍스처 샘플 수와 레이어 수가 비용입니다. Mix를 줄이고 채널 패킹(R=러프니스, G=메탈릭, B=AO 한 장에)으로.

디버그 테크닉

  1. 의심 구간 출력을 Emission에 직결해 값을 색으로 직접 보기 — 마스크가 의도대로인지 즉시 확인.
  2. Math > Greater Than으로 특정 값 범위만 하이라이트 — "어디서 1을 넘는가".
  3. 노멀 문제는 Geometry > Normal을 Emission으로 — 무지개색이 고르지 않으면 노멀 오염.

게임 이식 관점

블렌더 프로시저럴 노드는 게임 엔진으로 못 갑니다 — 최종적으로 텍스처로 베이킹해야 합니다. 처음부터 "이 재질은 베이크 가능한 구조인가"(오브젝트 좌표 의존, 4D 노이즈 등 베이크 불가 요소 회피)를 따지는 것이 전문가의 차이입니다.

48색 관리 — Filmic, AgX, ACES 전문가

"내 화면과 남의 화면이 다르다"를 끝내는 단계. 색이 어떻게 기록되고 표시되는지 통제합니다.

핵심 개념

  • 씬 리니어: 블렌더 내부 계산은 선형 공간 — 빛의 물리량 그대로.
  • 뷰 트랜스폼: 그 물리량을 화면용으로 바꾸는 변환. Standard(왜곡 없음, 클리핑 잘 됨) / Filmic / AgX(기본값, 밝은 영역이 자연스럽게 굴러떨어짐).
  • 텍스처 컬러스페이스: 색 텍스처는 sRGB, 데이터 텍스처(노멀/러프니스/마스크)는 반드시 Non-Color — 이거 하나로 노멀 맵 문제 절반이 설명됩니다.

실무 규칙

  1. 중간 산출물(컴포지팅으로 넘길 렌더)은 OpenEXR 32bit로 — 뷰 트랜스폼이 굽지 않은 리니어 데이터 보존.
  2. 최종 납품 직전에만 출력 공간으로 변환 (웹용 sRGB PNG 등).
  3. 외부 스튜디오 협업이면 ACES 같은 공통 규격 합의 — OCIO 설정 파일을 주고받습니다.
  4. 게임 에셋 텍스처는 뷰 트랜스폼 영향 없이 베이크되는지 확인 — 베이크는 Raw 데이터입니다.
!!

AgX로 보면서 만든 재질을 Standard로 보내거나 그 반대면 색이 완전히 달라집니다. 프로젝트 시작 시 뷰 트랜스폼을 팀 전체가 고정하세요.

49협업과 납품 규격 전문가

외주, 팀 작업, 클라이언트 납품 — 작업물이 아니라 신뢰를 납품하는 방법.

납품 규격서(스펙) 만들기

의뢰를 받든 주든, 시작 전에 한 장으로 합의합니다:

  • 기하: 폴리곤 예산(예: 소품 ≤3k 트라이), 단위/스케일, 피벗 위치 규칙, 적용해야 할 모디파이어.
  • 텍스처: 해상도, 채널 패킹 규칙, 네이밍, 컬러스페이스.
  • 파일: 포맷(FBX/GLB), 좌표계(Y-up/Z-up), 포함물(LOD? 콜리전 메시?).
  • 검수 기준: 어떤 뷰어/엔진에서 확인할지 — "내 블렌더에선 됐는데"를 차단.

리뷰 사이클

  1. 블로킹(실루엣만) → 1차 컨펌 → 디테일 → 2차 컨펌 → 텍스처/마감 → 납품. 단계마다 컨펌을 받아 되돌림 비용을 최소화.
  2. 피드백은 스크린샷에 그려서 주고받기 — 말은 휘발됩니다.
  3. 수정 횟수와 범위를 계약에 명시 (예: 단계별 2회) — 무한 수정 지옥 방지.

인수인계 가능한 파일

내가 사라져도 파일이 살아있게: 외부 텍스처 Pack 또는 상대경로 정리(File > External Data), 쓰지 않는 데이터 제거(Orphan Data Purge), 씬에 README 텍스트 블록(어디를 만지면 무엇이 바뀌는지) 포함.

50포트폴리오와 다음 단계 전문가

기술의 끝은 결과물로 증명하는 것. 50챕터의 마지막은 보여주는 법입니다.

포트폴리오 원칙

  • 최고 3점 > 평범한 30점. 보는 사람은 가장 약한 작품으로 실력을 추정합니다.
  • 과정을 함께: 와이어프레임, 텍스처 브레이크다운, 노드 그래프 한 장 — "어쩌다 나온 결과"가 아님을 증명.
  • 실시간 에셋이면 트라이 수/텍스처 규격 명기 — 게임 분야는 예산 준수 능력이 곧 실력.
  • ArtStation + 짧은 턴테이블 영상이 표준. 로블록스 쪽이면 실제 플레이 가능한 쇼케이스 맵이 최강의 포트폴리오.

전문가 루틴

  1. 릴리즈 노트 읽기 — 블렌더는 버전마다 워크플로가 바뀝니다. 새 기능을 한 달 안에 한 번은 실전 투입.
  2. 분해 연습 — 잘 만든 작품을 보면 "몇 개의 패스/레이어로 만들었나"를 역산하는 습관.
  3. 도구 제작 — 같은 일을 세 번 하면 애드온/노드그룹으로. 만든 도구가 곧 전문성의 증거.
  4. 공유 — 튜토리얼이나 에셋을 공개하면 피드백이 성장을 가속합니다.
TIP

여기까지 왔다면 더 이상 강좌가 필요한 단계가 아닙니다. 공식 매뉴얼과 릴리즈 노트, 그리고 마감이 있는 실전 프로젝트가 가장 좋은 스승입니다.
