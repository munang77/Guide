Luau 전문가 — 라이브 게임 운영

ECS, 커스텀 리플리케이션, 서버 권위 판정, 대규모 데이터, 라이브 운영. 상용 게임을 만들고 굴리는 진짜 전문가의 영역.

10챕터 리플리케이션 대규모 데이터 라이브 운영

41데이터 지향 설계와 ECS 전문가

객체 1만 개에서 OOP가 무너지는 지점 — 데이터와 동작을 분리하는 설계.

왜 데이터 지향인가

  • OOP는 객체마다 동작이 흩어져 캐시 효율이 나쁘고, "불타는 + 얼어붙은 + 날아가는" 같은 조합 폭발에 상속이 못 버팁니다.
  • ECS: Entity(그냥 ID) + Component(순수 데이터 테이블) + System(같은 컴포넌트 조합을 한 번에 처리하는 함수).
-- 개념 골격
local positions, velocities = {}, {}   -- 컴포넌트 저장소 [id] = data

local function moveSystem(dt)          -- 시스템: 일괄 처리
    for id, vel in velocities do
        positions[id] += vel * dt
    end
end
  • "불탄다"는 Burning 컴포넌트를 붙이는 것 — 어떤 엔티티든 즉시 조합 가능. 상속 트리가 사라집니다.
  • 일괄 루프는 병렬화(Actor)와도 궁합이 좋습니다.
  • 실전에서는 Matter, ECR 같은 검증된 Luau ECS 라이브러리 — 단, 게임 전체를 ECS로 할 필요는 없습니다. 대량·동질 객체(투사체, 군중, 도트뎀)에만 적용하는 하이브리드가 현실적.

42커스텀 리플리케이션 전문가

자동 복제에 맡기지 않고 — 무엇을, 누구에게, 언제 보낼지 직접 설계합니다.

상태 동기화 계층 만들기

  1. 서버 진실 저장소: 게임 상태를 인스턴스가 아닌 순수 테이블로 관리 (인스턴스는 그 표현일 뿐).
  2. 스냅샷 + 델타: 입장 시 전체 스냅샷 1회, 이후엔 바뀐 키만 (델타) 주기 전송.
  3. 관심 관리: 플레이어별로 "보이는 영역"의 엔티티만 동기화 — 거리 밖 정보는 보내지도, (치터가 볼 수 있으니) 노출하지도 않음.
  4. 우선순위 큐: 가까운 것/전투 중인 것 먼저, 멀리 있는 장식은 늦게.
-- 델타 수집 골격
local dirty = {}                       -- 이번 틱에 바뀐 것
local function setState(id, key, value)
    state[id][key] = value
    dirty[id] = dirty[id] or {}
    dirty[id][key] = value
end
-- 0.1초마다: 플레이어별 관심 엔티티의 dirty만 buffer로 패킹해 전송

클라이언트는 받은 상태로 표현(보간 포함)만 합니다. 이 구조까지 오면 수백 엔티티 실시간 게임도 대역폭이 통제됩니다 — 초고급의 buffer(38)와 네트워크 최적화(33)가 여기서 합쳐집니다.

43서버 권위와 히트 검증 전문가

반응성(클라가 즉시)과 공정함(서버가 판정) 사이의 전문가적 타협점.

3박자 구조

  1. 클라 예측: 입력 즉시 자기 화면에서 공격 모션/이펙트 — 체감 지연 0.
  2. 서버 판정: 같은 공격을 서버가 재계산(서버 좌표 기준 Raycast/거리) — 진짜 데미지는 여기서만.
  3. 화해(reconcile): 서버 결과가 예측과 다르면 클라가 보정 (빗나갔는데 맞은 척했던 이펙트 취소).

지연 보상 (Lag Compensation)

  • 클라가 본 화면은 과거입니다 — 서버는 각 캐릭터의 최근 위치 히스토리(0.5~1초)를 링버퍼로 보관.
  • 공격 패킷의 타임스탬프 시점으로 되감아서 판정 — "내 화면에선 분명 맞았는데"를 해결.
  • 되감기 한도를 두세요(예: 300ms) — 무한 신뢰는 치트 벡터가 됩니다.

검증 휴리스틱

검사기준실패 시
사거리무기 사거리 + 이동 여유 + 지연 여유무시
발사 각도바라보는 방향과 표적의 각도차무시
벽 관통서버 Raycast로 시야 확인무시 + 카운트
연사 속도무기 쿨다운의 95% 미만 간격무시 + 카운트
누적 의심카운트 임계 초과로그 → 그림자 모니터링 → 제재

오탐을 전제로 — 즉시 킥이 아니라 점수화·로그·사후 분석이 전문가의 제재 설계입니다.

44대규모 데이터 설계 전문가

동시접속 수천, 서버 수백 대에서 데이터가 깨지지 않게 — 세션 잠금과 트랜잭션.

세션 잠금 (Session Locking)

  • 문제: 유저가 서버 이동 시 두 서버가 같은 데이터를 동시에 들고 저장 — 늦게 저장한 쪽이 덮어씀 = 아이템 복사/소실 버그의 근원.
  • 해법: 데이터에 lock = {serverId, timestamp}를 UpdateAsync로 기록. 다른 서버가 잠근 데이터는 잠시 대기 후 재시도, 오래된 잠금(크래시 잔재)은 만료 처리.
  • ProfileService/ProfileStore 같은 검증된 라이브러리가 이 문제의 사실상 표준 — 원리를 알고 쓰는 것과 모르고 쓰는 것의 차이가 장애 대응에서 갈립니다.

설계 규칙

  • UpdateAsync만: 읽기-수정-쓰기를 원자적으로. SetAsync는 마지막 쓰기가 이깁니다.
  • 스키마 버전: data.version + 마이그레이션 함수 체인 (v1→v2→v3) — 옛 유저 데이터가 언제 로드돼도 안전.
  • 분리 저장: 자주 바뀌는 것(인벤토리)과 거의 안 바뀌는 것(도감)을 다른 키로 — 충돌 표면 축소.
  • MemoryStore: 서버 간 실시간 공유(매치메이킹 큐, 글로벌 경매)는 DataStore가 아닌 MemoryStore — 빠르고 휘발성.
  • 감사 로그: 고가 거래는 별도 OrderedDataStore에 기록 — 복구와 어뷰징 조사의 근거.

45Open Cloud와 외부 연동 전문가

게임 밖에서 게임을 조작하는 공식 통로 — 자동 배포, 외부 대시보드, 웹훅.

Open Cloud API로 할 수 있는 것

  • DataStore API: 웹에서 유저 데이터 조회/수정 — 고객지원 도구("아이템 복구해주세요")를 게임 밖에 구축.
  • Publishing API: CI에서 플레이스 자동 배포 — Git에 머지되면 테스트 유니버스에 자동 퍼블리시.
  • MessagingService API: 외부에서 전체 서버에 메시지 — 웹 관리자 페이지의 "공지 발송" 버튼.

게임 내부 ↔ 외부 다리

-- HttpService: 게임에서 외부 API 호출 (서버 전용, Game Settings에서 허용)
local HttpService = game:GetService("HttpService")
local ok, res = pcall(function()
    return HttpService:PostAsync(WEBHOOK_URL, HttpService:JSONEncode({
        event = "rare_drop", user = player.UserId,
    }))
end)
  • MessagingService(게임 내): 서버끼리 실시간 방송 — 크로스 서버 이벤트, 글로벌 공지.
  • 보안 수칙: API 키는 절대 게임 코드에 넣지 않습니다 — 키는 외부 서버에만, 게임은 자기 웹훅 주소로 보내기만. 외부 응답도 신뢰하지 않고 검증.
  • 속도 제한(쿼터)이 있으므로 호출은 큐에 모아 배치로.

46텔레메트리와 데이터 기반 운영 전문가

"감"이 아니라 측정으로 게임을 고칩니다 — 무엇을 기록하고 어떻게 읽을 것인가.

기록할 것

  • 퍼널: 입장 → 튜토리얼 완료 → 첫 전투 → 첫 구매 — 어느 단계에서 이탈하는가.
  • 밸런스: 무기별 사용률/승률, 스테이지별 클리어율, 평균 클리어 시간.
  • 경제: 코인 발생량 vs 소비량 — 인플레이션 조기 경보.
  • 기술: 평균 FPS(클라 보고), 서버 프레임 시간, 에러 발생률.

구현 계층

  1. 1차: AnalyticsService(공식) — 퍼널/이코노미 이벤트를 Creator Hub 대시보드로.
  2. 2차: 커스텀 이벤트를 HttpService로 자체 수집기에 — 게임 고유 지표.
  3. 이벤트 설계 규칙: 이름은 category_action(shop_purchase), 스키마 문서화, 개인정보는 UserId까지만.

A/B 테스트

-- UserId 해시로 결정적 분기 — 같은 유저는 항상 같은 그룹
local group = (player.UserId % 100 < 50) and "A" or "B"
local tutorialVersion = (group == "A") and "short" or "guided"

한 번에 한 변수만, 충분한 표본이 모일 때까지, 그리고 결과가 가설과 달라도 데이터를 따르는 것 — 운영의 규율입니다.

47라이브 게임 운영 전문가

출시는 끝이 아니라 시작. 무중단 업데이트, 핫픽스, 장애 대응의 체계.

업데이트 전략

  • 구버전 서버 공존: 퍼블리시해도 기존 서버는 구버전으로 계속 돕니다 — 데이터 스키마는 항상 한 버전의 하위 호환 유지.
  • 소프트 셧다운: 구서버 플레이어를 새 서버로 점진 이주 — 전원에게 안내 후 TeleportService로 재배치. 즉시 킥은 최후 수단.
  • 기능 플래그: 신기능을 코드에 넣되 꺼진 상태로 배포 → 일부 서버만 켜서 관찰 → 전체 활성화. 문제 시 코드 롤백 없이 플래그만 끄기.
-- 기능 플래그: MemoryStore/DataStore에서 주기 로드
if Flags.get("NewBossEnabled") then
    spawnNewBoss()
end

장애 대응 체계

  1. 감지: 에러율/저장 실패율 임계치 초과 시 웹훅 알림 (챕터 45).
  2. 완화 먼저: 원인 분석보다 피해 차단 — 문제 기능 플래그 오프, 거래 일시 중지.
  3. 복구: 감사 로그(챕터 44) 기반 데이터 복원.
  4. 사후 분석: 타임라인 기록, 재발 방지책을 테스트로 박제. 책임 추궁이 아니라 시스템 개선.
TIP

운영의 황금률: 금요일 저녁에 배포하지 않는다. 그리고 모든 변경은 되돌릴 방법을 먼저 정한 뒤에 배포한다.

48Luau 내부 동작 이해 전문가

왜 이 코드가 빠른가를 VM 수준에서 설명할 수 있는 단계.

알아두면 코드가 달라지는 내부 사실

  • 테이블 = 배열부 + 해시부: 1..n 연속 정수 키는 배열부(빠름), 그 외는 해시부. 구멍 난 배열(중간 nil)은 둘 다 망칩니다 — #연산도 신뢰 불가.
  • 네이티브 벡터: Vector3는 힙 할당 없는 내장 타입 — 수만 개 생성해도 GC 부담이 적습니다. 좌표 묶음에 테이블 대신 Vector3를.
  • 인라인 캐싱과 단형성: 같은 자리에서 항상 같은 모양의 테이블을 다루면 빠릅니다 — 같은 필드 구성을 유지하고, 필드를 nil로 지웠다 다시 만들기보다 처음부터 모든 필드를 초기화.
  • 업밸류: 클로저가 잡은 지역 변수 접근은 전역(_G, 게임 트리 탐색)보다 빠릅니다 — 핫루프에서 local find = table.find 캐싱이 통하는 이유.
  • 문자열 인터닝: 같은 문자열은 메모리에 한 번 — 비교는 빠르지만 연결(..) 반복은 매번 새 문자열 = GC 압박.

벤치마크 규율

local function bench(name, n, fn)
    local t = os.clock()
    for i = 1, n do fn() end
    print(string.format("%s: %.3fms", name, (os.clock()-t)*1000))
end

마이크로 최적화는 항상 측정과 함께 — 그리고 프로파일러가 가리키는 핫스팟에만. 나머지 코드의 우선순위는 성능이 아니라 가독성입니다.

49코드 리뷰와 팀 표준 전문가

전문가의 코드는 혼자 잘 쓴 코드가 아니라, 팀이 빨리 읽고 안전하게 바꿀 수 있는 코드입니다.

팀 표준 문서 (있어야 할 최소)

  • 스타일: 네이밍(camelCase 변수 / PascalCase 타입·클래스), 파일 구조, strict 의무화 범위. — 도구(StyLua 포매터, Selene 린터)로 자동 강제.
  • 금지 목록과 이유: 예) RemoteFunction의 서버→클라 Invoke 금지(클라가 안 돌려주면 서버 영원 대기), wait() 대신 task.wait().
  • Remote 추가 절차: 이름 규칙, 검증 체크리스트(초고급 34) 통과 의무.

리뷰 문화

  1. 작은 PR — 변경 300줄 이하. 리뷰 불가능한 크기는 그 자체가 결함.
  2. 리뷰 관점 우선순위: ① 보안(검증 빠짐?) ② 데이터 안전(저장 경로?) ③ 누수(연결 정리?) ④ 설계 ⑤ 스타일(도구가 할 일).
  3. 코멘트는 코드에, 제안은 대안과 함께 — "왜 이렇게 했어요?"보다 "X 경우에 nil이 될 수 있어 보여요, FindFirstChild는 어떨까요".
  4. 리뷰는 가르침의 통로 — 신규 멤버의 첫 한 달 리뷰 밀도가 팀의 1년을 결정합니다.

Rojo 기반(초고급 40)이면 GitHub PR 워크플로가 그대로 적용됩니다 — 브랜치, 자동 테스트, 승인 후 머지, 머지 시 자동 배포(챕터 45)까지가 전문가 팀의 파이프라인입니다.

실력 증명 방법

  • 오픈소스 라이브러리: 직접 만든 유틸(정리 도구, 네트워킹 래퍼)을 Wally에 배포 — 문서, 테스트, 버전 관리까지 갖추면 그 자체가 포트폴리오.
  • 실서비스 지표: "동접 X에서 서버 프레임 Yms 유지", "데이터 사고 0건 Z개월" — 전문가 이력서는 숫자로 말합니다.
  • 기술 글: 장애 사후 분석, 최적화 과정을 공개 — 설명할 수 있는 지식만이 진짜 내 것입니다.

계속 성장하는 루틴

  1. Luau RFC와 릴리즈 노트 구독 — 언어 자체가 계속 진화합니다 (네이티브 코드젠, 새 타입 기능).
  2. 다른 게임 분해: 잘 만든 게임의 네트워크 동작을 관찰하고 구조를 역설계해보는 습관.
  3. 가르치기: 커뮤니티 질문에 답하고 주니어를 리뷰 — 가르치다 막히는 곳이 자기 구멍입니다.
  4. 경계 넓히기: 일반 게임서버(Go/Rust), DB, 분산 시스템 — Roblox 밖의 지식이 Roblox 안의 설계를 한 단계 올립니다.
